# ADR-017: 수익 재투자 방식 및 드로다운 관리

## 상태

`결정됨`

## 날짜

2026-03-25

## 맥락

수익이 발생했을 때 포지션 크기를 키울지(복리), 초기 자본 기준을 유지할지(단리) 결정해야 한다.
또한 누적 드로다운에 대한 단계별 대응을 확정해야 한다.

## 결정

### 수익 재투자: 단리 (초기 자본 기준 고정)

```
포지션 크기 기준 = 초기 자본 (고정)
  → 수익이 나도 포지션 크기 불변
  → 손실이 나도 포지션 크기 불변

예시:
  초기 자본 10,000 USDT
  수익 +500 → 잔고 10,500 → 포지션 크기: 10,000 × 5% = 500 USDT
  손실 -300 → 잔고 9,700 → 포지션 크기: 10,000 × 5% = 500 USDT
```

왜 단리인가:
- **Paper 트레이딩 단계에서 전략 성능을 순수하게 평가**하려면 포지션 크기가 일정해야 한다
- 복리는 수익 구간에서 포지션이 커지면서 한 번의 큰 손실로 이전 수익을 모두 반납할 위험
- 단타 + 신규상장의 높은 변동성 환경에서 복리는 리스크 증폭
- 전략이 충분히 검증된 후(Live 안정화 이후) 복리 전환을 검토

### 초기 자본 재설정

Paper/Live 전환 시, 또는 의도적으로 자본 규모를 바꿀 때만 초기 자본을 재설정한다.

```yaml
# config/strategy/asset.yaml
capital:
  mode: fixed                     # fixed(단리) | compounding(복리)
  initial_balance: 10000          # 초기 자본 (USDT) — Paper 모드에서 사용
  # Live 모드에서는 최초 실행 시 Bybit 잔고를 initial_balance로 기록
```

### 드로다운 관리: 3단계 대응

드로다운 = (초기 자본 - 현재 잔고) / 초기 자본 × 100

```
[1단계] 드로다운 5% (잔고 9,500)
  → 텔레그램 경고 알림
  → 거래 계속

[2단계] 드로다운 10% (잔고 9,000)
  → 긴급 알림
  → 포지션 크기 50% 축소 (5% → 2.5%)
  → 리스크도 50% 축소 (1% → 0.5%)
  → 거래 계속 (축소된 규모)

[3단계] 드로다운 15% (잔고 8,500)
  → 긴급 알림
  → 모든 거래 즉시 중단
  → 열린 포지션은 SL/TP로 자연 종료 (신규 진입만 중단)
  → 수동 재개 필요 (config 변경 또는 명령)
```

왜 기존(10/15/20%)보다 타이트한가:
- 단타 + 1% 리스크에서 15% 드로다운이면 약 15회 연속 풀손절에 해당
- 이 수준이면 전략 자체에 구조적 문제가 있다고 판단해야 함
- 빨리 멈추고 전략을 점검하는 것이 올바른 대응
- Paper 모드에서는 드로다운이 발생해도 실제 돈이 아니므로 보수적 기준이 전략 검증에 유리

### 드로다운 회복 시

```
2단계 진입 후 드로다운이 5% 이하로 회복:
  → 포지션 크기 원복 (5%)
  → 알림: "드로다운 회복, 정상 규모 복원"

3단계 진입:
  → 자동 회복 없음
  → 수동으로만 재개 가능
```

### 설정

```yaml
# config/strategy/asset.yaml
capital:
  mode: fixed
  initial_balance: 10000

drawdown:
  warning_pct: 5                  # 1단계: 경고
  reduce_pct: 10                  # 2단계: 포지션 50% 축소
  reduce_factor: 0.5              # 축소 비율
  stop_pct: 15                    # 3단계: 거래 중단
  auto_recover: true              # 2단계에서 회복 시 자동 원복
  # 3단계는 수동 재개만 가능
```

## 영향

- 포지션 사이징: 항상 `initial_balance` 기준 계산
- 드로다운 계산: 매 거래 체결 후 갱신
- Notification: 드로다운 단계 진입/회복 알림
- Live 전환 시: Bybit 실제 잔고를 initial_balance로 기록하는 초기화 로직 필요
- 3단계 진입 시 수동 재개 인터페이스 필요 (config 수정 또는 텔레그램 명령)
