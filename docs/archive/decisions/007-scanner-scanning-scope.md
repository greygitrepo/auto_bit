# ADR-007: 스캐너 스캐닝 범위 및 데이터 수집 방식

## 상태

`결정됨`

## 날짜

2026-03-25

## 맥락

Scanner Strategy는 전체 시장에서 거래 대상 종목을 선정해야 한다.
Bybit Linear(USDT 무기한 선물)에는 200+ 심볼이 존재하며, 모든 심볼의 캔들 데이터를 실시간 수집하는 것은 리소스 부담이 크다.
Data Collector(P1)와 Scanner의 역할 분담 및 데이터 수집 범위를 결정해야 한다.

## 선택지

### 선택지 A: 2단계 스캐닝 (REST 1차 필터 → WebSocket 2차 상세)

**설명:**

```
[1차 필터 - 가벼운 REST API 호출]
  Bybit GET /v5/market/tickers (1회 호출로 전체 심볼 티커 조회)
  → 24h 거래량, 가격변화율, 펀딩비 등 기본 지표로 필터링
  → 50개 후보 심볼 선정

[2차 필터 - WebSocket 상세 수집]
  후보 심볼만 캔들/오더북 WebSocket 구독
  → 지표 계산 (EMA, RSI, MACD 등)
  → 최종 5개 거래 대상 선정
```

**장점:**
- 리소스 효율적 (WebSocket 구독 50개 vs 200+)
- REST 티커 API는 1회 호출로 전체 심볼 데이터를 가져옴 (Rate Limit 부담 없음)
- 1차 필터가 빠르게 불필요한 심볼을 제거
- WebSocket 구독 심볼을 동적으로 관리 가능

**단점:**
- 1차 필터에서 걸러진 심볼의 기회를 놓칠 수 있음
- 2차 필터 진입 시 캔들 히스토리 로드 지연 (최초 구독 시)
- 구독 심볼이 바뀔 때 WebSocket 재구독 관리 필요

### 선택지 B: 전체 심볼 데이터 수집 후 필터링

**설명:**

```
전체 200+ 심볼의 캔들 데이터를 WebSocket으로 수집
→ 모든 심볼에 대해 지표 계산
→ 스캐너가 전체 데이터에서 필터링
```

**장점:**
- 모든 심볼을 동등하게 분석 → 기회 누락 최소
- 구독 관리가 단순 (한번 구독하면 유지)
- 과거 데이터 축적으로 분석 정밀도 높음

**단점:**
- WebSocket 200+ 채널 구독 → 메모리/CPU 부하
- 지표 계산 대상이 200+ → P2 프로세스 부하
- 대부분의 심볼은 거래 대상이 되지 않으므로 리소스 낭비
- Bybit WebSocket 연결당 구독 제한 가능

### 선택지 C: REST 전용 스캐닝 (WebSocket은 선정 후에만)

**설명:**

```
[스캐너 - REST API만 사용]
  전체 심볼 티커 + 최근 캔들 데이터를 REST로 조회
  → 지표 계산 및 필터링
  → 최종 거래 대상 5개 선정

[Data Collector - 선정된 심볼만]
  선정된 심볼만 WebSocket 구독
  → Position Strategy에 실시간 데이터 제공
```

**장점:**
- WebSocket 구독을 최소화 (5개 심볼)
- 리소스 사용 가장 적음
- 스캐너와 데이터 수집의 관심사가 명확히 분리

**단점:**
- REST 캔들 조회 시 Rate Limit 주의 (심볼당 1회 호출)
- 50개 심볼 × 캔들 조회 = 50회 API 호출 → Rate Limit 내 가능하지만 여유롭지 않음
- 실시간성이 떨어짐 (REST 폴링 주기에 의존)

## 결정

**선택지 A 변형: 2단계 스캐닝 (신규상장 + 거래량/변동성 기반)**

스캐닝 대상을 "신규상장 종목" 중심으로 한정하고, 거래량과 변동성 순으로 30개 후보를 선정한다.

### 스캐닝 흐름

```
[1차 - REST API 풀 구성]
  Bybit GET /v5/market/instruments-info → 전체 심볼 + 상장일 조회
  Bybit GET /v5/market/tickers → 전체 심볼 24h 거래량/변동성
  → 신규상장 종목 필터 (상장 N일 이내)
  → 거래량 + 변동성 기준 정렬
  → 상위 30개 후보 풀 구성

[2차 - 후보 풀 상세 분석]
  30개 후보의 캔들 데이터 REST 조회
  + BTC/ETH 캔들 데이터 (상시 수집, 추세 판단용)
  → 지표 계산 + BTC/ETH 추세 반영 스코어링
  → 상위 종목 선정 (빈 슬롯 수만큼)
```

### 핵심 변경사항
- **스캔 대상:** 전체 시장이 아닌 "신규상장 종목" 풀로 한정
- **후보 풀:** 30개 (거래량 + 변동성 상위)
- **BTC/ETH는 상시 수집:** 스캐너 스코어링에 추세 데이터 반영
- **스캐너는 온디맨드 실행:** 포지션 빈 슬롯이 있을 때만 동작 (ADR-010 참조)

### WebSocket 구독 범위
- **상시 구독:** BTC/USDT, ETH/USDT (추세 판단 + 기준 데이터)
- **동적 구독:** 선정된 최대 3개 종목 (Position Strategy용 실시간 데이터)
- 나머지 30개 후보는 스캔 시점에 REST로 조회

## 영향

- Data Collector(P1): BTC/ETH 상시 WebSocket + 선정 종목 동적 구독
- 스캐너: REST API 기반 1차 필터, P2에서 2차 분석
- 신규상장 판별 기준(N일)을 config로 관리 필요
- Rate Limit: 1차(2회 호출) + 2차(30개 캔들 조회) = ~32회, 충분히 여유
