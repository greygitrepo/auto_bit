# ADR-006: 백테스팅 접근 방식

## 상태

`결정됨`

## 날짜

2026-03-25

## 맥락

전략의 과거 성과를 검증하기 위한 백테스팅 시스템이 필요하다.
외부 프레임워크를 사용하거나 자체 구현할 수 있다.

## 선택지

### 선택지 A: 자체 구현 (Paper 엔진 재사용)

**설명:**
Paper Trading 엔진을 과거 데이터에 대해 순차 실행하는 방식으로 백테스팅을 구현한다.

**장점:**
- Paper/Live와 동일한 로직 → 일관성 보장
- 코드 재사용 최대화
- 커스텀 요구사항에 유연하게 대응
- 추가 의존성 없음

**단점:**
- 성능 최적화 직접 해야 함
- 시각화, 리포트 직접 구현
- 개발 시간 소요

### 선택지 B: backtrader 사용

**설명:**
Python 백테스팅 프레임워크 backtrader를 사용한다.

**장점:**
- 검증된 백테스팅 엔진
- 다양한 분석 도구, 시각화 내장
- 커뮤니티, 문서 풍부

**단점:**
- 자체 전략 구조와의 어댑터 필요
- 학습 비용
- Paper/Live 전략 코드와 이중 관리 가능성
- 유지보수가 활발하지 않음

### 선택지 C: vectorbt 사용

**설명:**
벡터화 기반 고속 백테스팅 라이브러리.

**장점:**
- 매우 빠른 실행 속도 (numpy 벡터 연산)
- 파라미터 최적화에 적합
- 시각화 지원

**단점:**
- 이벤트 기반 전략과 구조가 다름
- 복잡한 전략 표현의 한계
- 학습 곡선

## 결정

**백테스팅 시스템을 구현하지 않는다.** 전략 검증은 **Paper Trading**으로 수행한다.

이유:
- 메인넷 실시간 데이터 기반 Paper Trading이 실전과 동일한 환경에서의 검증을 제공한다.
- 백테스팅은 과거 데이터에 대한 과적합(overfitting) 위험이 있으며, Paper Trading은 이 문제를 원천적으로 회피한다.
- 백테스팅 시스템 개발에 투입할 리소스를 핵심 기능(전략 파이프라인, Paper 엔진)에 집중한다.
- Paper Trading으로 실시간 전략 성과를 측정하면서 점진적으로 파라미터를 조정하는 방식이 더 실용적이다.

### 검증 워크플로우
```
1. 전략 개발/수정
2. Paper 모드로 실행 (메인넷 실시간 데이터)
3. 성과 모니터링 (승률, 손익비, 드로다운 등)
4. 파라미터 조정
5. 충분한 거래 샘플 확보 후 Live 전환
```

## 영향

- 프로젝트 범위 축소: `backtest/` 디렉토리 불필요
- 전략 코드가 Paper/Live 단일 구조로 유지 (이중 관리 없음)
- 검증에 실시간 시간이 소요되나, 실전 환경 정확도가 높음
- 파라미터 최적화는 Paper 운영 중 수동 조정으로 수행
