# ADR-004: 주문 방식 전략

## 상태

`결정됨`

## 날짜

2026-03-25

## 맥락

실제 주문 실행 시 시장가/지정가 선택, 슬리피지 관리, SL/TP 설정 방식을 결정해야 한다.
이 결정은 체결 확실성, 비용, 리스크에 직접 영향을 준다.

## 선택지

### 선택지 A: 시장가 위주 + API SL/TP

**설명:**
진입/청산 모두 시장가(Market)로 실행하고, SL/TP는 Bybit 서버사이드 조건부 주문으로 설정한다.

**장점:**
- 즉시 체결 보장
- 구현 단순
- SL/TP가 서버에 있으므로 클라이언트 장애에도 안전

**단점:**
- 슬리피지 발생 (특히 유동성 낮은 종목)
- Taker 수수료 (0.06%)

### 선택지 B: 지정가 위주 + 클라이언트 SL/TP 모니터링

**설명:**
진입은 지정가(Limit)로 시도하고, SL/TP는 클라이언트에서 가격 모니터링 후 시장가로 청산한다.

**장점:**
- Maker 수수료 절감 (0.01%)
- 정확한 진입가 보장

**단점:**
- 미체결 리스크 (가격이 지정가를 스쳐지나갈 수 있음)
- 클라이언트 장애 시 SL/TP 미작동 위험
- 구현 복잡도 증가

### 선택지 C: 하이브리드

**설명:**
진입은 지정가 시도 → 타임아웃 시 시장가 전환. SL/TP는 서버사이드 조건부 주문 사용.

**장점:**
- 수수료 절감 시도 + 체결 보장
- SL/TP 안전성 확보

**단점:**
- 구현 복잡도 증가
- 진입 타이밍 지연 가능

## 결정

**선택지 A: 시장가 위주 + API SL/TP** 채택.

이유:
- 즉시 체결이 보장되어 진입/청산 타이밍을 놓치지 않는다.
- SL/TP를 Bybit 서버사이드 조건부 주문으로 설정하므로, 클라이언트(P3 프로세스)가 장애 나더라도 포지션이 보호된다. 멀티프로세스 아키텍처에서 프로세스 장애 시 안전성이 핵심이다.
- 구현이 단순하여 초기 개발 속도를 확보한다.
- Taker 수수료(0.06%)는 전략 수익률로 커버 가능한 수준이다.

### 주문 실행 흐름
```
진입: Market Order → 즉시 체결
  └→ 체결 확인 후 SL/TP 조건부 주문 설정 (서버사이드)

청산:
  1) SL/TP 자동 발동 (서버사이드) — 정상 케이스
  2) 전략 청산 신호 → Market Order — 전략 판단에 의한 청산
```

## 영향

- 주문 실행: Market Order + 서버사이드 Conditional Order (SL/TP)
- 수수료: Taker 0.06% (진입/청산 모두)
- Paper 엔진: 시장가 시뮬레이션 + 슬리피지 모델링
- 안전성: 클라이언트 장애 시에도 SL/TP 보호 확보
