# ADR-002: 데이터 저장소 선택

## 상태

`결정됨`

## 날짜

2026-03-25

## 맥락

시스템은 캔들 데이터, 거래 이력, 포지션 정보, 성과 데이터를 저장해야 한다.
초기에는 단순하게 시작하되 향후 확장을 고려해야 한다.

## 선택지

### 선택지 A: SQLite

**설명:**
파일 기반 경량 DB. 별도 서버 불필요.

**장점:**
- 설치/운영 간편 (파일 하나)
- Python 표준 라이브러리에 포함
- 개발/테스트 환경에 적합
- 충분한 성능 (초당 수천 건 쓰기)

**단점:**
- 동시 쓰기 제한 (WAL 모드로 완화 가능)
- 대규모 데이터 시 성능 한계
- 원격 접근 불가

### 선택지 B: PostgreSQL

**설명:**
풀 스펙 관계형 DB.

**장점:**
- 강력한 동시성 지원
- 시계열 확장(TimescaleDB) 가능
- 원격 접근/모니터링 용이
- 프로덕션 검증됨

**단점:**
- 별도 서버 필요 (Docker 또는 설치)
- 초기 설정 비용
- 단일 사용자 시스템에 과할 수 있음

### 선택지 C: SQLite + Redis

**설명:**
영구 저장은 SQLite, 실시간 캐시는 Redis.

**장점:**
- 실시간 데이터(현재가, 포지션) 빠른 접근
- 영구 데이터는 SQLite에 안전하게 저장

**단점:**
- 두 시스템 운영 필요
- 초기 단계에서 과한 복잡도

## 결정

**선택지 A: SQLite** 채택.

이유:
- 단일 사용자 자동거래 시스템에 SQLite의 성능은 충분하다. 초당 수천 건 쓰기가 가능하며, 캔들 데이터 수집 빈도(초당 수십 건)를 넉넉히 처리한다.
- 멀티프로세스 환경에서의 동시 쓰기는 **WAL 모드**를 활성화하여 해결한다. WAL 모드는 읽기/쓰기 동시 접근을 허용한다.
- Python 표준 라이브러리에 포함되어 있어 추가 설치/운영 부담이 없다.
- DB 파일 하나로 백업/이관이 간편하다.
- 추후 데이터 규모가 커지면 PostgreSQL로 마이그레이션할 수 있으나, 현 단계에서는 과한 복잡도이다.

### 운영 방침
- WAL 모드 활성화 (`PRAGMA journal_mode=WAL`)
- 프로세스별로 별도 connection 사용 (멀티프로세스 안전)
- 캔들 데이터는 주기적으로 오래된 것 정리 (retention policy)

## 영향

- 배포 환경: 추가 DB 서버 불필요, 파일 기반
- 데이터 접근: 표준 `sqlite3` 모듈 사용, ORM 없이 직접 쿼리
- 백업: DB 파일 복사로 완료
- 멀티프로세스: 각 프로세스가 독립 connection, WAL 모드 필수
