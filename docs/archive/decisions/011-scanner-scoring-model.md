# ADR-011: 스캐너 스코어링 모델

## 상태

`결정됨`

## 날짜

2026-03-25

## 맥락

스캐너는 필터링을 통과한 심볼들에 점수를 부여하고 순위를 매겨야 한다.
스코어링 모델의 구조와 요소별 가중치를 결정해야 한다.

## 스코어링 요소

### 거래량 (Volume)

| 지표 | 설명 | 측정 방법 |
|------|------|----------|
| 24h 거래대금 | 절대적 유동성 | Bybit 티커 API `turnover24h` |
| 거래량 변화율 | 거래량 급증 감지 | 현재 24h vs 이전 24h |
| 상대 거래량 | 평균 대비 이상치 | 현재 거래량 / 20일 평균 거래량 |

### 모멘텀 (Momentum)

| 지표 | 설명 | 측정 방법 |
|------|------|----------|
| 가격 변화율 | 단기 방향성 | 1h, 4h, 24h 변화율 |
| RSI 위치 | 과매수/과매도 거리 | RSI(14) 중심(50)으로부터 거리 |
| EMA 정렬 | 추세 강도 | EMA20 > EMA50 > EMA200 정렬 여부 |

### 변동성 (Volatility)

| 지표 | 설명 | 측정 방법 |
|------|------|----------|
| ATR% | 평균 변동폭 비율 | ATR(14) / 현재가 × 100 |
| 볼린저 밴드 폭 | 변동성 수렴/확산 | (상단 - 하단) / 중간 |

### 선물 고유 (Derivatives)

| 지표 | 설명 | 측정 방법 |
|------|------|----------|
| 펀딩비 | 롱/숏 편향 | Bybit API `fundingRate` |
| 미결제약정 변화 | 포지션 유입/유출 | OI 24h 변화율 |

## 선택지

### 선택지 A: 고정 가중치

**설명:**
각 카테고리의 가중치를 설정 파일에 고정한다.

```yaml
scoring:
  weights:
    volume: 0.35
    momentum: 0.30
    volatility: 0.20
    derivatives: 0.15
  min_score: 60
```

**장점:**
- 구현 간단, 직관적
- config로 가중치 조정 가능
- 결과 해석이 쉬움

**단점:**
- 시장 상황에 따른 동적 조절 불가
- 최적 가중치를 경험적으로 찾아야 함

### 선택지 B: 카테고리별 독립 필터 (점수 없이)

**설명:**
점수를 매기지 않고, 각 카테고리별 최소 조건(threshold)만 설정한다.
모든 조건을 통과한 심볼을 선정하고, 거래대금 순으로 정렬한다.

```yaml
filters:
  volume:
    min_24h_turnover_usdt: 50000000
    min_volume_change_pct: 20
  momentum:
    rsi_range: [30, 70]
    min_price_change_1h_pct: 0.3
  volatility:
    atr_pct_range: [0.5, 5.0]
sort_by: turnover_24h_desc
```

**장점:**
- 가장 단순한 구현
- 각 필터 조건이 명확
- 디버깅/검증 용이

**단점:**
- 미묘한 차이를 구분 못함 (통과/탈락 이분법)
- 경계값 근처 심볼이 불안정하게 진입/탈락

### 선택지 C: 티어(Tier) 기반

**설명:**
각 지표를 티어(상/중/하)로 분류하고, 상위 티어가 많은 심볼을 우선 선정.

```
거래량 Tier: 상(상위 10%) / 중(상위 30%) / 하(그 외)
모멘텀 Tier: 상 / 중 / 하
변동성 Tier: 적정(상) / 과소·과대(하)

선정 기준: "상" 2개 이상 + "하" 0개
```

**장점:**
- 이상치에 강함
- 카테고리 간 스케일 차이 문제 없음

**단점:**
- 티어 경계값 설정 필요
- 세밀한 순위 구분 어려움

## 결정

**선택지 A: 고정 가중치 점수 기반** 채택. BTC/ETH 추세를 시장 환경 점수로 반영한다.

### 스코어링 구조 (0~100점)

```
종합 스코어 = 거래량(30%) + 변동성(25%) + 모멘텀(20%) + 신규상장(10%) + 시장환경(15%)
```

#### 거래량 점수 (30%)

```
점수 = normalize(24h_turnover_rank, 0~100) × 0.5
     + normalize(relative_volume, 0~100) × 0.5

relative_volume = current_volume / avg_volume_20d
  < 1.0  → 0점
  1.0~2.0 → 선형 0~50
  2.0~5.0 → 선형 50~100
  > 5.0  → 100점
```

#### 변동성 점수 (25%)

```
ATR% = ATR(14) / close × 100

적정 변동성 범위 (sweet spot):
  ATR% < 0.5  → 0점 (너무 낮음)
  0.5~1.0     → 선형 0~60
  1.0~3.0     → 선형 60~100 (최적)
  3.0~5.0     → 선형 100~50 (과변동 감점)
  > 5.0       → 0점 (탈락)
```

#### 모멘텀 점수 (20%)

```
RSI 점수:
  RSI < 20 or RSI > 80 → 0점 (극단, 진입 위험)
  RSI 20~30 → 40점 (과매도 반등 기대)
  RSI 30~50 → 60점 (상승 여력)
  RSI 50~70 → 80점 (추세 중)
  RSI 70~80 → 50점 (과매수 접근)

가격 변화율 점수:
  abs(price_change_4h) 기반, 방향은 시장환경과 일치 시 가점
```

#### 신규상장 점수 (10%)

```
상장 경과일:
  1~3일   → 100점
  4~7일   → 80점
  8~14일  → 60점
  15~21일 → 40점
  22~30일 → 20점
  > 30일  → 탈락 (후보 풀에서 제외)
```

#### 시장 환경 점수 (15%) — BTC/ETH 추세 반영

```
BTC 추세 판단 (1h 기준):
  EMA20 > EMA50 > EMA200 → 강세 (+1)
  EMA20 < EMA50 < EMA200 → 약세 (-1)
  그 외                    → 중립 (0)

ETH 추세 판단: 동일 로직

시장 방향:
  BTC + ETH 모두 강세 → 강세장 (bull)
  BTC + ETH 모두 약세 → 약세장 (bear)
  그 외              → 혼조 (mixed)

시장환경 점수:
  bull + 종목 상승 모멘텀  → 100점
  bear + 종목 하락 모멘텀  → 100점 (숏 기회)
  bull + 종목 하락 모멘텀  → 30점 (역추세)
  bear + 종목 상승 모멘텀  → 30점 (역추세)
  mixed                   → 50점
```

### 진입 불가 필터 (스코어링 후 적용)

스코어가 높아도 아래 조건에 해당하면 제외:

```
- 이미 포지션 보유 중인 심볼
- 동일 심볼 최근 손절 후 쿨다운 중 (기본 4시간)
- 24h 거래대금이 전일 대비 50% 이상 감소 (유동성 이탈)
- 스코어 < min_score (기본 55점)
```

### 스캐너 출력

```python
@dataclass
class ScanResult:
    symbol: str
    score: float              # 종합 스코어 (0~100)
    market_direction: str     # "bull" | "bear" | "mixed"
    suggested_side: str       # "LONG" | "SHORT" | "NEUTRAL"
    scores_detail: dict       # 카테고리별 점수 (디버깅용)
    reason: str               # 선정 사유 요약
    metadata: dict            # 추가 데이터 (거래량, ATR% 등)
```

`suggested_side`는 시장환경과 모멘텀을 종합한 방향 힌트로, Position Strategy에서 참고할 수 있다 (강제는 아님).

## 영향

- scanner.yaml에 가중치와 각 지표별 파라미터 관리
- ScanResult에 `market_direction`, `suggested_side` 추가 → Position Strategy에 컨텍스트 전달
- 디버깅: `scores_detail`로 종목 선정 사유를 투명하게 추적 가능
- 텔레그램 알림에서 선정 사유 표시 가능
