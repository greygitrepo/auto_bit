# ADR-021: GUI 프레임워크 및 구조

## 상태

`결정됨`

## 날짜

2026-03-25

## 맥락

거래 시작/종료, 현황 모니터링, 수익률 차트 등을 표시하는 GUI가 필요하다.
거래 엔진은 백그라운드에서 독립 실행되고, GUI는 이를 모니터링/제어하는 구조여야 한다.

요구사항:
- 거래 시작/종료 제어
- 거래 종목 및 현황 (현재 포지션, 미실현 P&L)
- 자산 현황 (잔고, 투입 자본, 유보 자본)
- 전체 자본 수익률 및 수익률 차트
- 종목별 수익률 및 수익률 차트
- 거래 시작 시 엔진은 백그라운드에서 실행, 별도 종료 신호 없으면 계속 동작

## 선택지

### 선택지 A: Web UI (FastAPI + React/Vue)

**장점:**
- 브라우저에서 접근, 원격 모니터링 가능
- 차트 라이브러리 풍부 (Chart.js, Recharts)
- 모바일에서도 접근 가능

**단점:**
- 프론트엔드 + 백엔드 이중 개발
- Node.js 빌드 환경 필요
- Python 프로젝트에 JavaScript 스택 혼합

### 선택지 B: Web UI (FastAPI + Jinja2 SSR)

**장점:**
- Python만으로 완결
- 별도 프론트엔드 빌드 불필요
- FastAPI와 자연스러운 통합

**단점:**
- 실시간 업데이트에 WebSocket 직접 구현 필요
- 차트 라이브러리는 여전히 JS (Chart.js 등)
- 인터랙티브한 UI 한계

### 선택지 C: 데스크톱 GUI (PyQt / PySide / Tkinter)

**장점:**
- Python 단일 스택
- 네이티브 성능
- 네트워크 불필요

**단점:**
- 원격 접근 불가
- 크로스 플랫폼 이슈
- 차트 라이브러리 제한적 (matplotlib은 실시간에 부적합)
- PyQt 라이선스 이슈 (GPL)

### 선택지 D: Streamlit

**장점:**
- Python만으로 완결, 코드량 최소
- 차트 내장 (Plotly)
- 빠른 프로토타이핑

**단점:**
- 커스텀 레이아웃 한계
- 실시간 업데이트 구조가 폴링 기반 (rerun)
- 프로덕션보다 프로토타입에 적합

## 결정

**선택지 B: FastAPI + Jinja2 SSR + WebSocket 실시간 업데이트** 채택.

이유:
- Python 단일 스택으로 완결 (프론트엔드 빌드 환경 불필요)
- FastAPI는 이미 WebSocket을 네이티브 지원 → 실시간 포지션/P&L 업데이트
- 차트는 Lightweight Charts (TradingView 오픈소스) 또는 Chart.js를 HTML에 임베드
- 로컬에서 브라우저로 접근하면서도, 향후 VPS 배포 시 원격 모니터링도 가능
- Streamlit보다 레이아웃 자유도 높고, React 대비 개발 복잡도 낮음

### 아키텍처

```
┌─────────────────────────────────────────────┐
│               Main Process                   │
│                                             │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐     │
│  │ P1 Data │  │P2 Strat │  │P3 Order │     │
│  └─────────┘  └─────────┘  └─────────┘     │
│        ↕ (Queue)                             │
│  ┌──────────────────────────────────────┐   │
│  │         Shared State (DB)            │   │
│  └──────────────────────────────────────┘   │
│        ↑ 읽기                                │
│  ┌──────────────────────────────────────┐   │
│  │      P5: GUI Server (FastAPI)        │   │
│  │        ├── REST API (제어/조회)       │   │
│  │        ├── WebSocket (실시간 푸시)    │   │
│  │        └── Static HTML/JS/CSS        │   │
│  └──────────────────────────────────────┘   │
└─────────────────────────────────────────────┘
         │
         ▼
   [브라우저: http://localhost:8080]
```

- GUI 서버(P5)는 별도 프로세스로 실행
- DB(SQLite)를 통해 거래 엔진의 상태를 읽기 전용으로 조회
- 거래 시작/종료 같은 제어 명령은 Main Process에 Queue로 전달
- 거래 엔진은 GUI 없이도 독립 실행 가능 (headless 모드)

### 실행 모드

```bash
# GUI와 함께 실행 (기본)
python -m src.main

# GUI 없이 백그라운드 실행
python -m src.main --headless

# GUI만 실행 (이미 돌고 있는 엔진에 연결)
python -m src.gui
```

### 거래 엔진 백그라운드 실행

```
[GUI에서 "거래 시작" 클릭]
  → Main Process에 start 신호 전달
  → 거래 엔진 프로세스들(P1~P3) 시작
  → GUI에 "실행 중" 상태 표시

[GUI 브라우저 닫기]
  → 거래 엔진은 계속 백그라운드 실행
  → 브라우저 다시 열면 현재 상태 표시

[GUI에서 "거래 종료" 클릭]
  → Main Process에 stop 신호 전달
  → 새 포지션 진입 중단
  → 열린 포지션은 SL/TP로 자연 종료 (옵션: 즉시 전체 청산)
  → 모든 포지션 청산 후 엔진 프로세스 종료
```

## 영향

- 프로젝트에 FastAPI, uvicorn, jinja2 의존성 추가
- `src/gui/` 모듈 추가
- 차트용 JS 라이브러리 (정적 파일)
- DB에서 읽기 전용 조회를 위한 API 엔드포인트 설계
- Main Process에 GUI 제어 Queue 추가
