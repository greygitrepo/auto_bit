## 2. 핵심 문제 진단

### 문제 1: 시그널 우선순위 없음 (가장 큰 임팩트)
- P3가 시그널을 큐 순서대로 처리 → 낮은 confidence 시그널이 먼저 포지션 슬롯 차지
- confidence 0.55짜리가 자리를 채운 후 0.97짜리가 "Max concurrent positions reached"로 거부
- 매 캔들마다 15~25개 시그널 중 5개만 진입 가능한데, 선착순으로 처리

### 문제 2: Short 방향 성과 극히 저조
- Long: 50건, 승률 38%, P&L -0.69
- Short: 54건, 승률 26%, P&L -1.33
- Short 손실이 Long의 약 2배, 전체 손실의 66% 차지

### 문제 3: Stop Loss 과다 (Phase 2)
- SL 42건(40%) → 총 -3.31 (전체 손실의 164%)
- ATR×1.5가 5분봉 스캘핑에서 너무 좁아 노이즈에 빈번히 걸림
- TP 도달률 16% (8건/50건) — R:R 1.5:1 대비 너무 낮음

### 문제 4: 트레일링 스탑 구조적 결함 (Phase 2)
- activation_r=0.6 → 수익 0.6R에서 조기 활성화
- trailing SL이 entry 아래에 설정되어 "수익 보호"가 아닌 "손실 발생" 초래
- 실제 사례: ASTERUSDT 트레일링 활성화 후 -0.0105 손실 청산

### 문제 5: 튜너 진동 (Phase 2)
- L0(3.3%) → L1(40%) → L0 → L1 무한 반복 (10회 기록)
- L0과 L1 사이에 적절한 중간 레벨 부재
- 실질적 파라미터 최적화 불가

## 3. 개선 계획

### Phase 1: 진입 품질 개선 (충돌 없음, 즉시 적용)

#### 1-1. 시그널 confidence 기반 우선순위 정렬
- 파일: `src/order/process.py`
- 변경: `_drain_signal_queue`에서 시그널을 배치로 수집 후 confidence 내림차순 정렬
- 기대: 고품질 시그널만 진입 → 승률 +10~15%p

#### 1-2. Short 방향 confidence 문턱 강화
- 파일: `src/order/process.py`
- 변경: Short 시그널에 더 높은 confidence threshold 적용 (예: 0.75 이상만 허용)
- 기대: 저품질 Short 진입 제거 → Short 손실 대폭 감소

### Phase 2: 청산 구조 개선 (Phase 1 결과 확인 후)

#### 2-1. SL/TP 비율 조정
- 파일: `config/strategy/position.yaml`
- 변경: ATR multiplier 1.5→2.0~2.5, min_pct 0.3→0.5, R:R 유지 또는 2.0으로
- 주의: Phase 1 이후 승률 변화를 확인하고 수치 결정

#### 2-2. 트레일링 스탑 보호 로직
- 파일: `src/strategy/position/momentum_scalper.py`
- 변경: trailing SL이 entry price 아래(LONG)/위(SHORT)로 내려가지 않도록 보장
- 주의: SL 거리 변경과 동시에 조정 필요 (activation_r, callback 연동)

#### 2-3. 튜너 점진적 조정 방식 개편
- 파일: `src/strategy/tuner.py`
- 변경: 레벨 점프 → 개별 파라미터 점진적 조정, 또는 수익 기반 튜닝
- 주의: DB 저장/복원, GUI 상태 표시 연쇄 확인 필요

### Phase 2 충돌 주의사항
- SL 넓히기 + 트레일링 수정은 반드시 동시 조정 (SL 넓히면 1R 커짐 → 트레일링 활성화 기준도 연동)
- 튜너는 Phase 1~2 이후 새로운 시그널 패턴에 맞게 재설계
